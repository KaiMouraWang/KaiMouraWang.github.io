Overview

Git is used for tracking changes in source code during software development.
It is designed for coordinating work amongst programmers, but it can be used to track changes in any set of files.

Basic/Typical Workflow

The basic workflow for using Git:

  1. Git clone - clone the source code you need to work on
git clone <url>
  1. Git log - check the recent changes made to the project
git log
  1. Git branch / checkout - create a new branch to work on so that you don‚Äôt mess up with the project
git branch <your branch name>
#or create branch and move to this branch immediately:
git checkout -b <your branch name>

You can use whatever editor you want; Git will track all the changes via checksums. Now that you made the necessary changes, it is time to put them on the staging area. As a reminder, the staging area is where you put modified codes that are ready to be snapshotted.

  1. Git add - Stage 
git add <filename>

You don‚Äôt have to add every file you modified to the staging area, only those which you want to be accounted in the snapshot. Now that the file is staged, it is time to ‚Äúcommit‚Äù it or putting its change in the database. We do this by using the command ‚Äúcommit‚Äù and attaching a little description with it.

  1. Git commit - commit the change to database, but only on your computer!
git commit
#or
git commit -m  "<message>"

Show the code to the senior dev first before making a push. If they are okay with it, you can merge your branch with the main snapshot of the project (called the master main branch). So first you must navigate back to the main branch by using the ‚Äúcheckout‚Äù command.

  1. Pull, Merge and Push:

6a) Switch to main branch

git checkout master/main

You are now on the master branch, where all the team‚Äôs work is stored. But the time you worked on your fix, the project may have changed, meaning that a team member may have changed some files. You should retrieve those changes before committing your own changes to master. This will limit the risk of ‚Äúconflicts‚Äù which can happen when two or more contributors change the same file.

6b) To get the changes, you have to pull the project from the remote server (also called origin ).

Git pull origin master/main

6c) Merge

Conflicts only arise when the same line has been modified by multiple people. If you and your coworkers changed different parts of the file, everything is okay. Now that we kept up with the current state of the project, it‚Äôs time to commit our version to master. You can merge your branch with the ‚Äúmerge‚Äù command.

git merge <your branch name>

6d) Push

Now that the commit has been merged back to master, it is time to push the changes to the main server. We do that by using to ‚Äúpush‚Äù command.

git push

Getting Started

Repositories

A repository is a storage where all your project and all the changes made to it are kept. You can think of it as a ‚Äúchange database.‚Äù But don‚Äôt worry; it is only a normal folder on your system, so it is very easy to manipulate.

Open Git Bash:

$ mkdir mynewproject
$ cd mynewproject/
$ git init

mkdir is a command used to create a directory; it is short for ‚Äúmake directory.‚Äù cd is the command used to navigate between directories; it is short for ‚Äúchange directory.‚Äù Finally, git init is short for ‚ÄúGit initialize.‚Äù

mkdir and cd are system commands; they are managed by the OS;
whereas init is a Git command. Every Git command begins with ‚Äúgit.‚Äù

Instead of tracking changes between versions, Git takes snapshots. All those snapshots are stored in the ‚Äú.git‚Äù directory. Each snapshot is called ‚Äúcommit‚Äù.
The HEAD file in this ‚Äú.git‚Äù directory points to the current ‚Äúbranch‚Äù or subversion of the project that you are working on. The default branch is called ‚Äúmaster main,‚Äù but it is just like any other branch; the name is just an old convention.

Working Directory

What about the empty area outside the ‚Äú.git‚Äù directory? It is called the Working Directory, and the files you will be working on will be stored there. Generally, your most recent version will be on the Working Directory.

Each file you work on is on the Working Directory. There is nothing particular about this place except the fact that you will only manipulate the files here directly. Never modify the files inside the ‚Äú.git‚Äù directory!

Git will detect any new file you will place in the Working Directory. And you check the status of the directory by using the Git command ‚Äústatus.‚Äù

Staging Area

The Staging Area is where your files go before the snapshots are taken.

$ git add <name_of_the_file>

Unstage a file using the Git command ‚Äú git rm ‚Äìcached ‚Äù

$ git rm --cached <name_of_the_file>

Don‚Äôt forget ‚Äò‚Äìcached‚Äò behind rm, otherwise you‚Äôll lose your file !!!

Commits

A commit is just a snapshot of the entire project at a certain time. Git doesn‚Äôt record the individual changes done to the files ; it takes a picture of the entire project .

The previous state of the project is another commit called ‚Äúparent.‚Äù The very first commit is created by Git when the repository is created, and it‚Äôs the one commit that has no parents. If a commit has two parents, that means that it was created by merging two branches.

A reference to a specific commit is called ‚Äúhead,‚Äù and it also has a name. And the head you are currently working on is called ‚ÄúHEAD‚Äù.

git commit

Staging a Change (git add)

Staging is the step that you must take before committing a change.
Staging is a feature in Git that enables the developer to choose what changes are actually going to get committed to the repository when the commit is made. There is a few ways you stage files:

# stage all files
git add --all

# stage all files (only if you are at the root of your project)
git add .

# stage selected files: git add [FILES]
git add file_1.txt file_2.txt
git add *.txt

Committing a Change (git commit)

When you make a commit to a Git repository, you are effectively ‚Äúsaving‚Äù the changes that you have staged to the repository.

Unlike saving files in most other programs, Git also requires a message to be saved against the commit along with some basic information about the user from the Git config shown above.

What you put in this message is important, so that you understand what it is that you changed on that particular commit.

‚Äòmessage‚Äô here is like a ‚Äòflag‚Äô or ‚Äòtag‚Äô, it is not the change you made in git add, it is just a reminder/sticker of what you have done.
